# VParticles – Stage 1 Development Guide

**Audience:** Junior developers joining the VParticles project

**Objective:** Build a working CPU particle simulation sandbox that establishes the architecture for the future VParticles engine.

---

# What is VParticles?

VParticles is **not** currently a Blender addon.

It is a **particle simulation engine** that may eventually:

* Become a Blender addon
* Become a standalone application
* Support GPU simulation
* Handle millions or billions of particles
* Be used for VFX, scientific visualization, and procedural effects

At this stage we are building the foundation.

---

# Stage 1 Mission

Create a simple application that:

```text
Creates particles
Updates particles
Displays particles
Keeps particles inside a window
```

Nothing more.

No GPU.

No Blender.

No physics.

No fancy UI.

Just a functioning simulation loop.

---

# Why We're Doing This

Many developers make the mistake of starting here:

```text
Blender Addon
 ↓
UI
 ↓
Particle Logic
```

We're doing the opposite:

```text
Particle Logic
 ↓
Renderer
 ↓
UI
 ↓
Blender Integration
```

The simulation engine is the valuable part.

Everything else is replaceable.

# Particle

A particle is just data.

```cpp
struct Particle
{
    float x;
    float y;

    float vx;
    float vy;
};
```

Meaning:

```text
x,y     = position
vx,vy   = velocity
```

Nothing else.

No rendering.

No logic.

Just data.

---

# ParticleSystem

The ParticleSystem owns all particles.

```cpp
class ParticleSystem
{
public:

    std::vector<Particle> particles;

    void Create(int count);

    void Update(
        float dt,
        float windowWidth,
        float windowHeight
    );
};
```

Responsibilities:

```text
Spawn particles
Update particles
Handle collisions
```

Responsibilities it DOES NOT have:

```text
Rendering
UI
Input
Window creation
```

---

# Application Layer

The main application is responsible for:

```text
Creating window
Reading keyboard input
Calling Update()
Drawing particles
```

This separation is important.

The particle engine should not know whether it is running in:

```text
SFML
OpenGL
Blender
Unreal
Unity
```

---

# Core Simulation Loop

Every frame:

```text
1. Measure elapsed time
2. Process input
3. Update particles
4. Render particles
5. Repeat
```

---

# Particle Movement

Every particle follows:

In code:

```cpp
p.x += p.vx * dt;
p.y += p.vy * dt;
```

Where:

```text
dt = time since last frame
```

This makes movement independent of FPS.

---

# Window Boundaries

Currently particles are confined to the window.

When a particle hits a wall:

```cpp
p.vx *= -1;
```

or

```cpp
p.vy *= -1;
```

This creates a bounce.

# Why Boundaries Exist

Without boundaries:

```text
Particle
 ↓
Moves forever
 ↓
Leaves screen
 ↓
Never seen again
```

Not useful for testing.

Boundaries keep all particles visible.

# What We Are NOT Building Yet

Do not implement:

```text
Gravity
Wind
Brownian Motion
GPU Compute
Multi-threading
Serialization
Blender Integration
```

These belong to later stages.

---

# Success Criteria for Stage 1

The build is considered successful when:

```text
✓ Window opens

✓ Particles render

✓ Particles move

✓ Particles bounce off walls

✓ Program runs continuously

✓ No crashes
```

# Final Rule for Stage 1 Team

Whenever you are unsure where to put code, ask:

> "Does this affect particle simulation, or does it affect presentation?"

If it affects simulation:

```text
ParticleSystem
```

If it affects display:

```text
Application Layer
```

Maintaining that separation from the beginning will make it much easier to evolve VParticles from a simple SFML experiment into either a Blender addon or a standalone particle engine in the future.
